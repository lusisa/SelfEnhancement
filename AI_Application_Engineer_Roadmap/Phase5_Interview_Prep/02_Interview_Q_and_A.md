# Phase 5: 面试冲刺核心 Q&A

整理了一些针对该 JD 极有可能被问到的高频面试题。

## 1. 大模型/框架底层原理篇

**Q1: LangChain 中的 Chain 和 Agent 有什么本质区别？**
*   **思路**：Chain 是硬编码的、确定性的工作流（Step 1 做 A，Step 2 做 B）。Agent 是动态的，由 LLM 作为大脑自主决定下一步调用什么工具（不确定性工作流）。

**Q2: 如何解决 RAG 中“大海捞针”导致的“幻觉”问题？**
*   **思路**：1. 提升检索质量：混合检索、Query 重写、加入 Rerank 模型。2. 提升语料质量：更好的分块策略、清洗脏数据。3. Prompt 限制：“仅根据参考资料回答，不知则答不知”。

**Q3: 解释一下 Tool Calling (Function Calling) 的实现机制？**
*   **思路**：我们把工具的名字、描述、参数类型 (JSON Schema 格式) 一起在 Prompt 中发给大模型。模型如果觉得需要用，会返回一个要求调用函数的特殊结构（包含参数），我们拿到后在本地执行代码，再把执行结果拼回去发给模型。

## 2. 工程与算法篇

**Q4: 如果大量用户同时调用你的 FastAPI AI 接口，会导致阻塞吗？如何解决？**
*   **思路**：回答异步编程的价值。因为请求大模型 API 是 I/O 密集型操作。在 FastAPI 中使用 `async def` 和 `await`，使得在等待 API 响应时，系统线程可以挂起去处理其他请求，实现高并发。

**Q5: 如果向量数据库检索特别慢，怎么排查优化？**
*   **思路**：从索引算法聊起。提一下 HNSW 索引，可能需要调整参数（如 m, ef_construction），牺牲一点准确率换取极大的速度提升。

## 3. 业务与软技能篇 (行为面试)

**Q6: 如果业务部门的人抱怨说你做的 AI 系统总是答非所问，你作为架构师会怎么处理？**
*   **思路**：不要直接回答修改代码。
    1.  **倾听与沟通**：与业务方开会，拿具体的 Bad Case 来分析。
    2.  **根因定位**：是知识库没录入相关数据？是检索没查到？还是大模型没总结好？
    3.  **技术方案**：建立埋点日志和反馈机制 (点赞/踩)，引入评估集系统 (Evaluation) 持续监控 RAG 准确率。

**Q7: 平时怎么关注行业前沿技术？**
*   **思路**：举一些具体的例子。比如关注 GitHub 上的爆款项目，看 HuggingFace 的模型榜单，看吴恩达的推特，或者是日常使用各种 AI 工具提升自己的效率。

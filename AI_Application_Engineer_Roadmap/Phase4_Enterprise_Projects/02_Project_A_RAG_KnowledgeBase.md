# 杀手锏项目 A: 汽车底盘设计规范 RAG 问答系统 (企业级知识库)

在求职时，一个高质量的 RAG 项目比十个 Toy Project 都有说服力。本项目的目标是向面试官展示你具备**处理复杂工业文档、优化 RAG 检索链路**的能力。

## 1. 项目业务背景
车企研发中心积累了数以万计的 PDF 设计规范、国家强制标准(GB)、材料测试报告。研发工程师在设计新零件时，每天要花大量时间去这些海量 PDF 中翻找某个具体的“公差范围”或“材料屈服强度”。
**你的目标**：做一个 AI 助手，工程师提问，AI 给出准确答案，并附上引用了哪份 PDF 的哪一页，坚决杜绝“幻觉”。

## 2. 技术栈架构
*   **后端**：FastAPI + Python
*   **核心框架**：LlamaIndex (比 LangChain 更擅长处理复杂 RAG 数据流) / LangChain
*   **向量数据库**：Milvus 或 Qdrant (使用 Docker 本地部署)
*   **大模型接口**：通义千问 API / 智谱 API / 或是本地部署的 Ollama (如 qwen2-7b-instruct)
*   **Embedding 模型**：bge-large-zh-v1.5

## 3. 面试级亮点设计 (如何与普通求职者拉开差距？)

如果你只做了基础的 Load -> Chunk -> Embed -> Retrieve，面试官会觉得你只是跑了个 Demo。你需要在这个项目中实现以下 **1 到 2 个高阶特性**：

### 亮点一：复杂文档解析 (Parsing)
汽车规范中有很多**表格**和**图片**。普通的拆分策略会把表格拆得稀巴烂，导致大模型无法理解。
*   **技术方案**：使用专门的解析器 (如 LlamaParse, 或是阿里通义的文档解析服务)，将带有表格的 PDF 解析为 Markdown 格式，通过 Markdown 结构保持表格的完整性再进行索引。

### 2. 亮点二：混合检索 (Hybrid Search) + 重排 (Reranking)
对于工业词汇（如特定零件号 `BKT-559-X`），向量模型的语义检索能力很差，传统关键词匹配效果更好。
*   **技术方案**：在数据库端开启 Sparse 向量 (BM25) + Dense 向量 (Embedding) 双路召回。捞回 20 个结果后，引入 BGE-Reranker 模型打分重排，最终取前 3 名喂给 LLM。

### 3. 亮点三：文档层级索引 (Parent-Child Retriever)
不要直接检索小 Chunk，因为小 Chunk 缺乏上下文。
*   **技术方案**：切分大 Chunk 作为“父节点” (用于给 LLM 阅读)，再把大 Chunk 切成几个小 Chunk 作为“子节点” (用于向量检索)。搜到了小 Chunk，就把它所在的整个大 Chunk 提取出来。

## 4. 实施步骤建议
1.  **数据准备**：去网上下载 3-5 份真实的汽车国家标准文档 (如《乘用车制动系统技术要求.pdf》)。
2.  **构建 Pipeline**：写脚本将 PDF 解析、分块、存入你本地跑在 Docker 里的 Milvus。
3.  **接口开发**：用 FastAPI 写一个 `/chat` 接口，接收 Query，返回 Answer 和引用来源。
4.  **前端展示 (可选)**：用 Streamlit 或 Gradio 写一个极简的网页界面展示效果。

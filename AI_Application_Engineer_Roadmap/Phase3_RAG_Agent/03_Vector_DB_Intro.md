# Phase 3: Vector Database (向量数据库) 基础入门

关系型数据库（如 MySQL）通过精准匹配或模糊查询 (LIKE) 来找数据；而向量数据库通过**语义距离**来找数据。

## 为什么需要向量数据库？
大模型虽然聪明，但它的记忆是停留在训练截止日期的。要让它知道企业的私有规范文档，就需要用 RAG 方案。而 RAG 的数据存储载体，就是向量数据库。
一句话：**向量数据库是大模型的外挂长期记忆。**

## 主流向量数据库选型指南

在企业级落地中，选型是一门学问：
1.  **Chroma / FAISS (本地轻量级)**：适合个人学习、跑开源 Demo 或是数据量非常小（几万条以内）的项目。使用简单，甚至无需单独部署服务端，作为 Python 库直接运行。
2.  **Milvus / Qdrant (企业级部署)**：适合千万级到十亿级的向量检索，支持集群部署，高可用，支持元数据过滤 (Metadata Filtering)，是大型车企构建知识中心的首选。
3.  **Elasticsearch (pgvector / Redis) (混合级)**：在传统的数据库/搜索引擎上安装向量检索插件。如果公司原来就有庞大的 ES 集群，通常会选择这种方案以降低运维成本，也是工业界极常见的做法。

## 核心技术点 (面试必考)
*   **距离度量算法 (Distance Metrics)**：
    *   **Cosine Similarity (余弦相似度)**：最常用，衡量两个向量在空间中的夹角，不受向量长度影响。
    *   **L2 / Euclidean (欧氏距离)**：两点间的绝对距离。
    *   **Dot Product (内积)**：在向量均被归一化 (Normalized) 的前提下，计算效率极高，等同于余弦相似度。
*   **索引算法 (Indexing Algorithms)**：海量数据中如何做到毫秒级检索？不可能挨个对比。常用的近似最近邻搜索 (ANN) 算法是 **HNSW (Hierarchical Navigable Small World)**。
